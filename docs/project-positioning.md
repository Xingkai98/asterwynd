# 项目定位

## 目标

Asterwynd 的项目目标是服务于拿到大厂 Agent 相关开发岗位 offer。所有功能设计、技术取舍、文档组织、测试证据、benchmark 结果和面试叙事都必须服务于这个目标。

## 目标岗位

目标岗位以 Agent 相关开发为主线，同时覆盖 AI Infra、LLM 应用、RAG 和后端工程基本能力。

不把项目定位成纯算法、纯后端、纯前端或泛 AI 应用项目。Asterwynd 的核心价值应该体现为：能够设计、实现、验证和解释一个真实可运行的 Coding Agent 系统。

## 能力结构

Asterwynd 采用“一条主线 + 四个支撑面”的叙事结构。

**主线：Coding Agent 系统**

- Agent 运行时
- 工具调用协议
- 上下文管理
- 代码仓库理解
- 代码修改与验证
- 可观测性
- benchmark 闭环

**支撑面 1：AI Infra**

- 运行时边界
- 沙箱与权限
- 并发任务
- 可观测性
- 评测系统
- 资源与成本意识

**支撑面 2：LLM 应用**

- 多 provider 适配
- tool call 协议兼容
- 模型响应差异处理
- 上下文压缩
- prompt 与任务包装

**支撑面 3：RAG**

RAG 是后续增强方向，重点应该服务于代码仓库理解，例如 repo indexing、代码检索、上下文召回和任务相关文件定位。RAG 不能喧宾夺主，不能把项目带偏成知识库问答系统。

**支撑面 4：后端工程**

- FastAPI / WebSocket
- CLI
- 异步任务
- 配置与环境变量
- 日志
- 测试体系
- 工程结构

## 能力证明链

每个重要功能都要能回答：

- 它对应什么面试要求？
- 它解决什么真实 Agent 工程问题？
- 代码在哪里？
- 测试在哪里？
- benchmark 或运行证据是什么？
- 面试时如何讲清楚设计深度和取舍？

不能只堆功能列表。没有证据链的功能，不应该作为重点投入。

## 当前阶段判断

当前仓库已有 AgentLoop、工具系统、WorkspacePolicy、Hook、Memory、Skill、SubAgent、Trace、CLI/Web、Benchmark 等基础资产。主要问题不是从零重写，而是目标口径、功能阶段、文档结构和开发流程需要重新对齐。
